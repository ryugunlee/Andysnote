# AndysNote 개발 원칙

## 0. 이 앱의 개발 목적
이 앱은 개인이 구글 클라우드에 손쉽게 접근해서 글이나 문서를 쉬운 형식으로 남길 수 있기를 목적으로 만들어졌다.
이 앱은 글이나 문서를 폴더 등에 나누어 담아서 구조화하여 남겨 원하는 문서를 쉽게 찾고, 그 구조를 이해하기 쉽게 하기 위해 만들어졌다.
이 앱은 폴더와 캘린더를 통하여 쉽게 원하는 문서를 찾고, 본인이 쓴 글들을 기억하고 찾기 쉽게 하기 위해 만들어졌다.

## 1. 프로젝트 철학

AndysNote는 장기간 사용할 개인 노트 애플리케이션이다.

이 프로젝트는 다음의 가치를 가장 중요하게 생각한다.

- 단순함
- 유지보수성
- 확장성
- 예측 가능한 동작
- 사람이 읽을 수 있는 데이터

빠른 기능 추가보다 좋은 구조를 우선한다.

---

## 2. 저장 방식

AndysNote의 모든 문서는 .txt와 .md로 저장한다.

Markdown은 서식을 표현하는 유일한 문법으로 사용한다.

저장 형식을 독자적인 포맷으로 변경하지 않는다.

모든 문서는 메모장만으로도 읽을 수 있어야 한다.

---

## 3. 책임 분리

각 모듈은 하나의 책임만 가진다.

### Editor

담당

- 입력
- 커서
- 선택 영역
- 키보드 입력
- Undo / Redo

하지 않는 일

- Markdown 해석
- HTML 생성
- 저장 처리

---

### Markdown Engine

담당

- Markdown 문법
- Markdown 파싱
- Markdown 서식 추가
- 블록 타입 판별

하지 않는 일

- DOM 조작
- 저장
- 렌더링

---

### Renderer

담당

- 화면 표시
- 스타일 적용
- Live Preview

하지 않는 일

- Markdown 수정
- 저장
- 입력 처리

---

### Storage

담당

- Drive 저장
- Local 저장
- 자동 저장
- 캐시

하지 않는 일

- Markdown 처리
- 화면 렌더링

---

### Settings

담당

- 환경설정
- 사용자 옵션
- 테마
- 폰트

하지 않는 일

- 에디터 기능 구현
- 저장 로직 구현

---

## 4. 데이터 원칙

사용자가 작성한 텍스트가 항상 원본(Source of Truth)이다.

렌더링된 HTML은 임시 데이터이다.

Markdown은 저장 데이터이다.

화면은 언제든 다시 만들어질 수 있어야 한다.

---

## 5. 확장성

새로운 기능은 기존 구조를 최대한 수정하지 않고 추가할 수 있어야 한다.

예시

- 표
- 이미지
- 체크리스트
- 각주
- 수식
- 링크
- 플러그인

이러한 기능을 추가할 때 Editor를 크게 수정해서는 안 된다.

---

## 6. UI 철학

글쓰기에 집중할 수 있는 인터페이스를 만든다.

불필요한 애니메이션은 사용하지 않는다.

화려함보다 일관성을 우선한다.

---

## 7. 구조 원칙

파일 하나는 하나의 책임만 가진다.

모듈 간 결합도를 최소화한다.

명확한 책임을 가진 구조를 유지한다.

불필요한 추상화는 만들지 않는다.

---

## 8. 구현 원칙

기능을 만들기 전에 먼저 책임을 결정한다.

어떤 모듈이 담당하는지 먼저 정의한 후 구현한다.

구현 이후 구조를 맞추는 것이 아니라,
구조를 먼저 설계한 뒤 구현한다.

---

## 9. 글쓰기 우선

AndysNote는 메모 앱이 아니라 글쓰기 도구이다.

모든 기능은 글쓰기 경험을 향상시키기 위해 존재한다.

기술적으로 멋진 기능이라도 글쓰기 경험을 해친다면 다시 검토한다.

---

## 10. 기존 기능 보호

새로운 기능을 구현할 때는 기존 기능을 최대한 유지한다.

요청하지 않은 리팩토링은 하지 않는다.

관련 없는 코드를 수정하지 않는다.

기존 동작을 깨뜨리지 않는 범위에서 구현한다.

---

## 11. 장기 목표

AndysNote는 장기간 발전할 프로젝트이다.

단기적인 편의보다 장기적인 유지보수를 우선한다.

모든 기능은 앞으로 추가될 기능을 고려하여 설계한다.

작은 프로젝트처럼 구현하되,
큰 프로젝트처럼 구조를 생각한다.

## 12. 사용자가 주도한다.

AI는 개발을 돕는 도구일 뿐이다.

프로젝트의 방향은 항상 사용자가 결정한다.

AI는 구조를 제안하고 구현을 도와줄 수 있지만,
프로젝트의 철학과 설계 원칙을 임의로 변경해서는 안 된다.

새로운 기능이나 구조 변경이 필요한 경우에는 먼저 제안하고,
사용자의 승인을 받은 뒤 구현한다.

AI는 반드시 한국어로 말해야 한다.
